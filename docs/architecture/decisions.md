# Архитектурные решения (ADR)

Короткие записки о важных развилках. Для каждой — что решили и почему.

---

## ADR-001 · Модульный монолит

**Что:** один Spring Boot процесс, код раскладывается по пакетам-фичам (`auth`, `pet`, `vet`, `schedule`, `chat`).

**Почему:** микросервисы дают распределённую сложность (сетка, дискавери, distributed tracing) ради независимого деплоя — а нам деплоить нечего раздельно. Монолит проще разрабатывать, дебажить и хостить. Если модуль вырастет — отделим без переписывания бизнес-логики.

---

## ADR-002 · OpenAPI как источник правды

**Что:** контракты API живут в `api/specs/*.yaml`. Бэк генерирует Kotlin-интерфейсы и DTO из спек; контроллеры реализуют эти интерфейсы.

**Почему:** документация и код не расходятся. Изменение спеки → компилятор подсветит места, где сигнатура контроллера больше не соответствует. На фронте типы пока дублируем вручную в `frontend/src/lib/api.ts` — эндпоинтов мало, кодгенерация была бы лишней церемонией.

---

## ADR-003 · Spring Data JPA, не jOOQ

**Что:** доступ к БД через `JpaRepository<T, ID>` со стандартными методами по сигнатуре.

**Почему:** начали было с jOOQ ради типобезопасных запросов (PR #12 — closed), но для CRUD-операций он давал больше boilerplate, чем сэкономленных багов. JPA-репозитории читаются сами по себе (`findByOwnerIdOrderByCreatedAtDesc`), а сложные запросы при необходимости можно вытащить через `@Query`. Решено в PR #13.

---

## ADR-004 · JWT в `localStorage`, без refresh

**Что:** access-токен 24 часа, кладётся в `localStorage`, шлётся в `Authorization: Bearer`. Refresh-токенов нет.

**Почему:** короткий TTL access-токена нужен ради безопасности (если токен утёк), но добавляет код для рефреша. На MVP цена утечки невелика, упрощаем. Когда понадобится — добавим refresh в httpOnly cookie без переделок текущей схемы.

---

## ADR-005 · Чат через REST + polling, WebSocket позже

**Что:** фронт каждые 3 секунды дёргает `GET /api/conversations/{id}/messages`. Отправка через `POST`.

**Почему:** WebSocket требовал бы либо STOMP-эндпоинт + клиента, либо raw WS + протокол поверх, плюс обработку реконнектов. Polling работает с первого дня, легко дебажится, не зависит от сети с прокси. UI чата уже сделан так, чтобы переход на WS был незаметен — заменим источник сообщений и всё. PR #18 + #19.

## ADR-006 · Один прод-стенд, без testing-окружения

**Что:** в `Deploy` workflow один job `deploy`, который катит сразу в прод. До PR #26 был промежуточный `deploy-testing`, делал то же самое на той же VM.

**Почему:** два деплоя на одну машину — иллюзия безопасности. Реальное testing-окружение потребовало бы отдельной VM + отдельной БД + DNS-разводки. Пока команда из одного человека — флипаем сразу в прод, держим хорошие тесты и feature-флаги (когда понадобятся).

---

## ADR-007 · Self-hosted observability вместо облачного

**Что:** Prometheus + Grafana + Loki + Promtail на той же VM, добавлены в PR #27.

**Почему:** Grafana Cloud / Datadog дали бы тот же UX за подписку, но текущий стек влезает в ~250 МБ RAM и не зависит от внешних сервисов. ELK обсуждали и отвергли: Elasticsearch один съел бы 2-4 ГБ. Loki индексирует только лейблы, а не контент — для нашего объёма полнотекстовый поиск можно делать grep'ом в `LogQL`.

## ADR-008 · Forest-дизайн (PR #25)

**Что:** глубокая зелень `#16573e` + коралл `#ff6a4d` на мятно-белом фоне, Manrope + DM Sans, радиус 16. Никакого кремового и синего.

**Почему:** изначальный кремовый+терракотовый (v1) пользователь забраковал. Из трёх премиум-направлений в дизайн-хэндофе (Onyx / Bordeaux / Forest) выбрали Forest как самое читаемое и нейтральное для медицинской ниши. Не клиника, не корпоратив — приятный editorial-вид.

---

## ADR-009 · `com.xvet` пакет остаётся

**Что:** ребрендинг в ИКС ВЕТ — только видимый. Внутри код остаётся в `com.xvet`, Docker-образы зовутся `xvet-backend`/`xvet-frontend`, GitHub-репо тоже `1xvet`.

**Почему:** массовое переименование пакетов — много места для опечаток, обновляет каждую импортную строку и роняет историю blame. Технические идентификаторы стабильны, бренд — это то, что пользователь видит. Решено в PR #26.
